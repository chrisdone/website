One step forward, two steps back

The issue is that programming languages don’t go forward, they move sideways or diagonally, or sometimes backwards.

A new car comes out, and it has some cool feature: Hey, it has road surface detection and changes the steering accordingly! But it should also come with all the old stuff that you come to expect. Comfy seats, seatbelts, airconditioner, heated windows, wipers, proximity detection, power steering, cruise control, etc.

With new programming languages, what you tend to get is a chassis, engine and steering wheel, and the road surface detection.

Here is a list of cool ideas that have been discovered and implemented in programming languages, but which do not in their whole make up any existing language:

  • Contracts
  • Pattern matching
  • Syntactic abstraction1
  • Static types
  • Type inference
  • Dependent types
  • Parametrized modules
  • Overloaded literals2
  • Fundamental immutability
  • Purity
  • Interpreter
  • Closures
  • Multiple dispatch
  • Inheritance
  • Garbage collection
  • A debugger
  • Well-supported concurrency
  • Parallelism
  • Dynamic scope3
  • Dynamic code update4
  • Image-based persistent
  • Infinite-sized integers and rationals
  • Exceptions
  • Explicit null
  • Generic equality
  • Restarts
  • A comprehensive platform / library set (e.g. Java, .NET)
  • REPL/interaction mode
  • Compiler—IDE integration
  • Interoperability (with C, Java, JavaScript, CLR)
  • Support domain-specific languages
  • Proofs
  • Distribution
  • A compiler that compiles to machine code, either via static compilation, JIT compilation, or third-party engines
  • Tacit programming
  • Value polymorphism

The moral is, perhaps if you’re making a new language, try building upon the past, rather than ignoring it and starting from scratch?

As a follow-up post I might make a matrix of the top, say, 30, general purpose programming languages and all the features that they tick off.
